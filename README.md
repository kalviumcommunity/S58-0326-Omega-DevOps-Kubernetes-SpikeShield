# Squad_58_Omega_SpikeShield

**PR2**:- *comprehensive overview of how Kubernetes orchestrates the entire lifecycle of a workload, shifting focus from basic container execution to the complex mechanics of production-level management. The text details how the system maintains a desired state through a hierarchical process, where deployments use ReplicaSets to ensure self-healing and consistent pod availability. It further explores the technical signals used to monitor health, such as probes and resource limits, which dictate whether a container is ready for traffic or requires a restart. Ultimately, the material serves as a diagnostic guide to help students interpret common failure states and understand the automated recovery logic that keeps applications stable during updates and hardware issues.*

**PR2**:- *structured framework for understanding the CI/CD pipeline execution model, shifting the focus from automation as a generic tool to a rigorous separation of responsibilities. By distinguishing between Continuous Integration (CI), which validates code through testing and artifact creation, and Continuous Deployment (CD), which manages the release of those pre-built artifacts, the text establishes a clear flow of ownership. The primary objective is to teach students how to reason about pipeline changes safely by identifying the unique boundaries between application logic, automation scripts, and the infrastructure that runs the code. Ultimately, the text argues that this modular approach is essential for predictable releases and reducing the risks associated with modern software production.*

**PR4**:- *It details the necessary configuration of a local DevOps workstation, ensuring that essential utilities like Git, Docker, kubectl, and Helm are correctly installed and verified. Participants must demonstrate their technical readiness by managing containerised applications and Kubernetes clusters directly from their terminal. To prove successful completion, students are required to document their environment through terminal screenshots and a comprehensive video demonstration. Finally, all evidence must be submitted via a public pull request, including a README file that specifies the operating system and tools utilised. This structured approach ensures every developer possesses a robust, cloud-native development environment before beginning their project builds.*
